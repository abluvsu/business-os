Architecture Deep Dive

Why I Rewrote BusinessOS: Multi-Tenant Architecture Lessons

How a local-first SQLite app became a Clerk-authenticated, tenant-scoped SaaS platform — and the three enforcement layers that make the isolation actually hold.

For the first several months of building BusinessOS, "architecture" meant one thing: a local SQLite file on disk, opened by a WorkspaceManager, read and written by whichever process happened to be running on the founder's machine. It worked. It was fast. It had exactly one user in mind — the person running it.

Then I decided to make it a SaaS product, and every one of those simplifying assumptions became a liability. This is the story of what actually broke, what I replaced it with, and the three decisions I'd defend in a design review.

The Old Shape: One File, One User, No Boundary

The original data layer looked like this:

Before — Local-First
graph TD
    CLI[Local BusinessOS Process]
    WM[WorkspaceManager]
    SQLite[(workspace.db — SQLite file on disk)]

    CLI --> WM
    WM -->|opens by file path| SQLite
    SQLite -->|owner, workspaceId, everything| CLI

There was no login. There was no tenant_id column. WorkspaceManager tracked an "active workspace" the way a text editor tracks an open file — a path on disk, a status flag, nothing more. Isolation between customers wasn't a database concern, it was a filesystem concern: your data was safe because it was your machine.

That's a completely reasonable architecture for a local-first tool. It is not an architecture that survives contact with a second user, let alone a thousand.

The Rewrite: Identity Becomes a First-Class Object

The core move wasn't "add a login page." It was: make every request carry a TenantContext, and make that context the only thing the data layer trusts.

After — Multi-Tenant SaaS
graph TD
    Browser[Next.js SaaS UI]
    Clerk[Clerk Auth — issues JWT]
    Fastify[Fastify API — context plugin]
    TC[TenantContext: userId, orgId, workspaceId, role]
    Repo[TenantRepository — scoped queries]
    DB[(Postgres + RLS, tenant_id on every table)]

    Browser -->|1. sign in| Clerk
    Browser -->|2. Bearer JWT| Fastify
    Fastify -->|3. verifyToken| Clerk
    Fastify -->|4. resolve or auto-provision| TC
    TC --> Repo
    Repo -->|5. every query WHERE workspaceId = ctx.activeWorkspaceId| DB

Three pieces make this work, and each one taught me something the tutorials don't cover.

1. Resolve identity once, in a preHandler hook — not per-route

Every route handler is tempting ground for an ad-hoc if (userId !== resource.userId) throw 403. That pattern scales badly: it means tenant isolation is only as strong as the least-careful route you wrote. I moved identity resolution into a single Fastify plugin (plugins/context.ts) that runs before every /api/* request, verifies the Clerk JWT, and attaches a fully-resolved TenantContext — userId, organizationId, activeWorkspaceId, role — to the request object. Route handlers never see a raw JWT. They see a context, or they don't get called.

The side effect I didn't plan for but ended up loving: this is also where auto-provisioning lives. First time a Clerk-verified user hits the API, there's no row for them yet — so the hook opens a transaction, creates users, organizations, memberships, and a default workspaces row atomically, and only then builds the context. No separate signup flow, no "finish setting up your account" screen. Signing in is onboarding.

if (userRows.length === 0) {
  await db.transaction(async (tx) => {
    await tx.insert(users).values({ id: userId, clerkUserId, email, ... });
    await tx.insert(organizations).values({ id: organizationId, ... });
    await tx.insert(memberships).values({ userId, organizationId, role: "owner" });
    await tx.insert(workspaces).values({ id: activeWorkspaceId, organizationId, ... });
  });
}

The trade-off: this hook now carries meaningful complexity — including a "self-repair" branch for users whose membership link is missing, which exists purely because I didn't want to write a one-off backfill migration for early test accounts. That's a deliberate hack, not an oversight, and it's the kind of thing I'd flag in review: it buys time now, and it should have an expiry date.

2. Push isolation down to the repository, not the route

Even with a trustworthy TenantContext, I didn't want isolation to depend on every future route author remembering to filter by workspace. So the context never reaches db directly — it's wrapped in a TenantRepository that's the only sanctioned way to touch tenant data:

async getKnowledgeSources() {
  return this.db
    .select()
    .from(schema.knowledgeSources)
    .where(eq(schema.knowledgeSources.workspaceId, this.context.activeWorkspaceId));
}

async insertKnowledgeSource(data) {
  return this.db.insert(schema.knowledgeSources).values({
    ...data,
    workspaceId: this.context.activeWorkspaceId,
    tenantId: this.context.userId,
  });
}

Notice what's missing from the caller's contract: there's no workspaceId parameter to forget or spoof. It's injected from the context the repository was constructed with. A route handler that wants another tenant's data literally cannot express that query — the repository's type signature doesn't allow it. That's the property I actually care about: the compiler and the constructor enforce isolation, not code review discipline.

3. Defense in depth: repository scoping and database-level RLS

The TenantRepository layer is the one I trust today, but I don't trust it alone. The SaaS spec lays out the second layer: every table carries a tenant_id, Postgres Row-Level Security is enabled per table, and policies enforce tenant_id = request.jwt.claims.sub at the database itself:

ALTER TABLE workspaces ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_policy ON workspaces
    FOR ALL
    USING (tenant_id = request.jwt.claims.sub);

This is the layer that saves you when the application layer has a bug — a missing .where() clause, a repository method someone bypassed with a raw query, a future contributor who didn't know the rule. Application-layer scoping is ergonomic and fast to write against; RLS is the backstop that doesn't care how careful your TypeScript was. I'd rather have both and rarely need the second one than have only the first and find out the hard way.

What I'd Do Differently

Two decisions I'm still watching:

Mock fallback mode. When CLERK_SECRET_KEY isn't set, the context plugin falls back to reading x-user-id / x-workspace-id headers directly and auto-provisioning the same way. This kept local development and CI zero-config — no Clerk account required to run tests. The risk is obvious: that branch has to be provably unreachable in production, forever, or it's an authentication bypass wearing a "local dev" label. Right now that guarantee lives in one if (clerk) check. It should probably live somewhere more load-bearing before this ships broadly.
SQLite-shaped schema, Postgres-shaped ambitions. The current TenantRepository runs against the same Drizzle schema whether the backing store is local SQLite (via node:sqlite or Turso/libSQL) or the target multi-tenant Postgres. That's deliberate — it let me ship the identity and scoping model before committing to infrastructure. But tenant_id as a nullable TEXT column on a SQLite table is not the same guarantee as an RLS-enforced column on Postgres, and I'm aware the migration isn't just a pg_dump.

The Actual Lesson

The rewrite wasn't really about swapping databases or adding a login screen — those are the parts that show up in a diagram. The real work was deciding where tenant identity gets to enter the system (one hook), how far it's allowed to travel unchecked (nowhere — it's wrapped before it touches data), and how many independent layers get to say no before a query runs (two: repository scoping, and RLS as backstop). Everything else was plumbing.
